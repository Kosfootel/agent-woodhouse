# Vigil Home — Email Alert Thresholds

## Severity → Email Mapping

| Severity   | Email Sent? | Condition |
|------------|-------------|-----------|
| **CRITICAL** | ✅ Yes | Suricata priority 1, trust < 0.2, or anomaly z ≥ 4.0 |
| **HIGH**     | ✅ Yes | Suricata priority 2, or anomaly z ≥ 3.0 |
| **MEDIUM**    | ❌ No  | Suricata priority 3 (logged to dashboard only) |
| **LOW**       | ❌ No  | Suricata priority 4 or DNS/HTTP errors (dashboard only) |
| **INFO**      | ❌ No  | Normal events, flows, netflows |

**Thresholds are configurable** via `EMAIL_SEVERITY_LEVELS` in `app/email_notifier.py`.

## Alert Types That Trigger Email

| Alert Type | Severity | Example Trigger | Email Sent? |
|------------|----------|-----------------|-------------|
| **New unknown device, suspicious behaviour** | CRITICAL | Device with unknown MAC exhibiting high-traffic patterns on first seen | ✅ Yes |
| **Port scanning detected** | CRITICAL | Suricata alert: port scan detected from external IP | ✅ Yes |
| **Trust score drops below 0.2** | CRITICAL | Bayesian trust model drops due to multiple anomalous events | ✅ Yes |
| **Multiple failed auth attempts** | HIGH | Suricata alert: repeated SSH/RDP auth failures | ✅ Yes |
| **Unusual outbound connections** | HIGH | Device connecting to known-bad IPs or unusual ports | ✅ Yes |
| **Traffic anomaly (z ≥ 3.0)** | HIGH | Statistical anomaly in traffic volume | ✅ Yes |
| **Large file transfer anomaly** | MEDIUM | Unusual data exfiltration pattern | ❌ No (dashboard) |
| **New device discovered** | LOW | First-time device seen on network | ❌ No (dashboard) |
| **DNS resolution failure** | LOW | DNS query returned non-NOERROR | ❌ No (dashboard) |
| **HTTP error (4xx/5xx)** | LOW | HTTP response ≥ 400 | ❌ No (dashboard) |

## Rate Limiting

| Parameter | Default | Configurable? |
|-----------|---------|---------------|
| Window | 1 hour (3600s) | `VIGIL_EMAIL_WINDOW` env var |
| Max per window | 10 emails | `VIGIL_EMAIL_MAX` env var |

When rate-limited, alerts are still created in the Vigil database — only the email is suppressed.

## Trust Score Ranges

| Score Range | Meaning | Action |
|-------------|---------|--------|
| 0.8 – 1.0 | Trusted device | No alert |
| 0.4 – 0.8 | Normal behaviour | Informational |
| 0.2 – 0.4 | Suspicious | High alert, email sent |
| 0.0 – 0.2 | Compromised | Critical alert, email sent immediately |

## How It Works

1. **Suricata** generates alerts in `eve.json`
2. **detection.py** consumer processes each line:
   - Maps Suricata priority 1-2 → critical/high
   - Runs through anomaly detector (z-score)
   - Updates Bayesian trust model
3. **Alert creation** triggers email if severity is critical/high:
   - AlertEmailContext is built with device info, trust score, narrative
   - Rate limiter is checked (max 10/hr)
   - Email is sent via configured SMTP provider
4. **email_poller.py** (optional) runs as separate container:
   - Polls Vigil API every 30s for new critical/high alerts
   - Acts as backup if inline email fails
   - Persists last-processed alert ID for restart resilience

## Email Template

```
Subject: [Vigil ALERT] CRITICAL: Port Scanning Detected

Vigil Home Security Alert
────────────────────────

Severity: CRITICAL
Time: 2026-05-07 22:00:00 UTC
Device: MacBook-Pro (AA:BB:CC:DD:EE:FF)
Alert Type: Port Scan

External host 10.0.0.99 scanning ports 22,80,443 on MacBook-Pro.
Anomaly detected: 999 connections in 60 seconds (z=12.4)

Trust Score: 0.2150
Device Type: computer

View in Dashboard: http://192.168.50.30:8000/devices/42

────────────────────────
Alert ID: 7
This alert was generated by Vigil Home on gx-10.local
```

HTML version is sent as multipart/alternative with severity-colored header.
